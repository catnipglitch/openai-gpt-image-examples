# ツール要件

## 1. プロジェクト概要
- **ツール名**: Catnip Image Generator
- **目的**: ユーザーが簡単に画像を生成・編集できるツールを提供する。

## 2. 機能要件
- 画像の生成（AIベース）
- 画像の編集（フィルター、トリミング、リサイズ）
- プリセットテンプレートの提供
- ファイルのエクスポート（PNG、JPEG、SVG）
- **複数のバリエーション画像を一度に生成可能**
  - ユーザーが指定した数のバリエーションを同時に比較検討できる
- **シード値の指定と再利用が可能**
  - 好みの生成結果を再現するためのシード値管理
  - APIがシード値をサポートしなくても内部で一貫性を保持
- **情報入力はJSON形式で受け付ける**
  - テストモードでは引数での入力もサポート
- **ログ出力はJSONおよびCSV形式で対応**
  - **ログにはシード値情報を含め、再現性を確保**
- **出力ログに画像生成の推定料金を表示**
  - **ドルと日本円（1USD=160JPY）で表示**
  - **テキストプロンプト料金は含まず**

## 3. 非機能要件
- ユーザーフレンドリーなUI
- セキュアなデータ保存
- **設定ファイルは使用しない**
- **APIキーは環境変数で管理**
- **実行環境はWindows 11のみをサポート**

## 8. ファイル出力仕様
- **ファイル名指定がない場合は自動生成**
  - **生成されたファイル名にはシード値を含める**
  - **形式例: `generated_[timestamp]_[seed].[extension]`**
- **同名ファイルが存在する場合は自動的にインデックスを付与**
  - **例: `filename.png` → `filename_1.png`**
- **ディレクトリ名とファイル名の両方が指定された場合、フルパス指定のファイル名を優先**
- **出力先ディレクトリの指定がない場合、実行ファイルと同じフォルダの下に`output`ディレクトリを作成**

## 9. ユーザー体験の重視ポイント
- **バリエーション比較機能でクリエイティブな意思決定を容易に**
  - 複数の生成結果をグリッド表示して視覚的に比較
  - お気に入りの結果をマークして後で参照可能
- **シード値を通じた一貫した体験**
  - 気に入った結果を再現する手段をユーザーに提供
  - 他のAI画像生成APIへの将来的な拡張も視野に入れた設計
- **直感的なバージョン管理**
  - ファイル名の自動生成とインデックス付与でファイル管理の手間を軽減
  - 生成履歴とその設定を後から参照・再利用可能

## 10. システムアーキテクチャと拡張性

### モジュール構成
システムは以下の主要モジュールに分割し、拡張性と保守性を確保する：

1. **入力処理モジュール**
   - JSON入力の解析と検証
   - コマンドライン引数のパース
   - プロンプト情報の正規化

2. **画像生成コアモジュール**
   - 生成リクエストの調整
   - バリエーション管理
   - シード値の処理と永続化

3. **プラットフォーム連携モジュール**
   - 各AI画像生成APIへのアダプター
   - OpenAI DALL-E対応
   - Google Gemini対応
   - Stable Diffusion対応
   - 将来的なプラットフォーム追加を容易にする抽象化レイヤー

4. **出力処理モジュール**
   - 画像ファイル保存
   - ファイル名生成と重複回避
   - ディレクトリ管理

5. **ログ・レポートモジュール**
   - JSON/CSVログ出力
   - 料金計算
   - 生成履歴の管理
   - パフォーマンス指標の記録

### プラットフォーム抽象化
各画像生成プラットフォーム(OpenAI, Gemini, Stable Diffusion等)は共通インターフェースを実装し、以下の機能を標準化：

- プロンプト変換
- シード値対応（APIがサポートしない場合は内部エミュレーション）
- エラーハンドリング
- レート制限対応
- 料金計算

### 拡張性への考慮
- **プラグイン方式**: 新しい画像生成APIは、共通インターフェースを実装するだけで追加可能
- **設定の一元管理**: 環境変数経由でAPIキーなどの設定を管理
- **APIバージョニング対応**: APIの変更に柔軟に対応できる設計
- **スケーラビリティ**: 処理の並列化や分散化が将来的に可能な設計

### データフロー
1. ユーザー入力 → 入力処理モジュール → 正規化されたリクエスト
2. リクエスト → 画像生成コアモジュール → プラットフォーム別処理の実行
3. 生成結果 → 出力処理モジュール → 保存された画像ファイル
4. 全プロセス → ログ・レポートモジュール → JSON/CSVログと料金情報

### 非同期処理とステータス管理
- **非同期画像生成**
  - 画像生成処理はブロッキング処理を避け、非同期で実行
  - 複数リクエストの並行処理に対応
  - 長時間かかる処理でもUIやCLIの応答性を維持

- **進行状況の可視化**
  - 各処理ステップのリアルタイムステータス更新
  - 処理フェーズの明示（初期化、リクエスト送信中、生成待機中、ダウンロード中、保存処理中など）
  - バリエーション生成時は全体の進捗率も表示

- **イベント駆動型アーキテクチャ**
  - 処理ステップ間の疎結合を実現するイベントシステム
  - 各プラットフォームからのコールバック処理の標準化
  - 失敗したリクエストの再試行機能

- **タイムアウト管理**
  - 各処理ステップの適切なタイムアウト設定
  - 接続問題や応答遅延の検出と対応
  - ユーザーへのタイムアウト通知とアクション選択肢の提供

この非同期処理設計により、ユーザーは長時間の画像生成処理中も常に現在の状況を把握でき、特にバッチ処理時の体験を向上させる。また、将来的なバックグラウンド処理やタスクキュー実装への拡張も容易になる。この分割設計により、特定のプラットフォームへの依存を最小限に抑え、将来的な機能拡張やAPI変更に柔軟に対応できる構造となる。
